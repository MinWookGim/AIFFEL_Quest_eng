# 최병철 — 영상→탐지 웹서비스 + 현실 검사기술 리서치 (우리 한계의 맥락)

> ※ 내가(발표자) 취합한 범위의 정리다. 서비스 내부 구현과 제작 과정은 최병철 것이 정본이고, 아래 일부는 화면 밖에서 자동으로 확인이 안 돼 최병철 설명과 보내준 자료에 기댔다.

최병철의 갈래는 **"이 문제를 현실에선 어떻게 푸는가"를 조사해, 우리 결과의 위치를 잡아주는 쪽**이다. ①돌아가는 탐지 서비스를 실제로 배포했고, ②현장 검사 기술(수동 검사·열화상)을 리서치해 우리 한계를 맥락 속에 놓았고, ③외부 데이터셋을 발굴해 김민에게 넘겨줬다.

## 1. 영상 → 탐지 웹 서비스 (Cloud Run 배포)

**영상 파일을 올리면 2초 단위로 프레임을 뽑아 오염·손상을 감지하는 웹 서비스**를 구글 AI Studio로 만들어 Google Cloud Run에 배포했다.

- 서비스 주소: https://video-dataset-creator-196080853339.asia-south1.run.app (접속 가능)
- 흐름: 영상 업로드 → 2초 간격 프레임 추출 → dirt / damage 탐지
- 의미: 아이디어를 **돌아가는 실물 서비스**까지 끌고 갔다. 발표에서 라이브 시연이 가능한 데모다. (단, 현장 영상은 우리 학습 도메인과 또 달라, 도메인갭을 감안해 되는 만큼 정직하게 보여주는 게 맞다.)

## 2. 현실 검사기술 리서치 — AS-IS(수동) vs TO-BE(열화상+AI)

최병철은 실제 풍력 블레이드 검사가 현장에서 어떻게 이뤄지는지 유튜브 사례를 모아 **AS-IS ↔ TO-BE** 대조로 정리했다.

**AS-IS — 사람이 직접 보고 수동으로 찾는다** (*EasyInspect*, https://www.youtube.com/watch?v=htY-zi3ihAo)
- 작업자가 고해상도 사진을 눈으로 보며 마우스로 이상 부위에 직접 박스·다각형을 친다.
- 결함 종류를 텍스트로 태깅하고 심각도(Severity)를 **주관적으로 판단**, 손상 크기도 점을 찍어 수동 측정한다.
- 메뉴에 "Automatic issue detection coming in 2018" = 그 시점엔 **100% 수동 도구**.
- 한계: 막대한 시간·비용, 작업자 숙련도·피로에 따른 **휴먼 에러**.

**TO-BE — 열화상으로 안 보이는 걸 본다** (*DAAS Group* https://youtu.be/BbNpIu5J9Ds, *WindVue* https://youtu.be/ux1vtPpFFsU)
- 드론+열화상 카메라로, 육안·RGB로 안 보이는 **내부 박리·수분 침투·미세 크랙**을 온도 차로 가시화한다.
- 추천 장면: `[02:01]` 일반 vs 열화상 1:1 비교, `[02:13]` 결함 스팟 집중, `[02:45]` 세로 스캔+거리(m) 눈금, `[02:52]` 전체 파노라마 매핑.

## 3. ★ 우리 결과와의 연결 — "도메인갭 붕괴"를 맥락에 놓기

우리 팀 공통으로 아팠던 대목은, 모델이 학습한 캐글(NordTank) 밖 외부 데이터(WTBD·blade30·영상)에선 성능이 **확 무너진** 점이다(도메인갭). 최병철의 리서치는 이 약점을 그냥 "우리 모델이 약하다"로 끝내지 않고, **왜 이 문제가 원래 어려운가**로 다시 읽게 해준다.

- **손상 종류마다 필요한 센서가 다르다.** 표면 오염·큰 손상은 드론 RGB 사진(우리 캐글 데이터)으로 볼 수 있지만, **미세 크랙·블레이드 내부 손상은 RGB로는 원천적으로 한계**라 열화상이 필요하다. 우리가 RGB 한 종류만 다뤘으니, 낯선 손상 유형 앞에서 무너지는 건 데이터 방식의 한계이기도 하다.
- **현장도 정확한 탐지는 아직 사람 손을 탄다.** 업계 도구조차 최근까지 사람이 직접 화면을 보며 수동으로 박스를 쳤다(휴먼 인 더 루프). 즉 이 문제는 단일 RGB 모델 하나로 자동 완결되는 난이도가 아니다.
- 그래서 우리 도메인갭 붕괴는 **"모델이 못났다"기보다 "이 문제가 멀티모달(RGB+열화상)+사람 검토를 요구하는 난제"**라는 신호로 읽을 수 있다. 일종의 정직한 방어이자, 다음 단계로 가는 이정표다.
- 최병철이 **AI Studio로 탐지 서비스를 만든 것**도 이 맥락 위에 있다 — 완전 수동(AS-IS)을 자동화(TO-BE) 쪽으로 한 걸음 옮겨보려는 시도.

## 발표 배치 제안

- **프로젝트 배경 / 기존 기술의 한계**: EasyInspect 수동 어노테이션 캡처 → "지금은 사람이 일일이 마킹한다"로 자동화의 필요성 강조.
- **도입부 후킹 또는 회고·로드맵**: 열화상 비교 장면(`02:01`) → "우리는 RGB만 봤고, 다음 단계는 열화상 예지보전".
- **비교분석·회고**: 우리 도메인갭 결과를 위 3번 논리로 재해석(약점 → 문제 난이도의 증거).
