# MQ02 심사위원 예상질문 방어집 (슬라이드 기준 우선순위판)

심사위원은 **슬라이드에 보이는 것**에서 질문을 만든다. 그래서 질문을 "어느 슬라이드가 유발하나" 기준으로
1순위(슬라이드가 직접 유발)와 2순위(자료를 깊게 읽은 사람용)로 나눴다.
형식: **질문 (유발 슬라이드) -> 30초 답변(그대로 말하면 되는 존댓말) -> 주머니(뒷받침 숫자·그림)**.
잣대 표기: A = 공식 채점기(model.val) / B = 자체 채점기(conf 0.05, 상대비교용).
페이지 번호는 최종 발표자료(slides_v3, 18장) 기준.
이 문서는 팀원들에게도 공유한다 — 심사위원이 발표자만 지목하리란 보장이 없어서, 맨 아래 "팀원별 갈래 질문"에 각자 받을 수 있는 질문과 답변 뼈대를 정리해뒀다.

# 1순위 — 슬라이드가 직접 유발하는 질문

## Q1. "동일 조건 자체 채점"이라고 썼는데, 자체 채점이 뭡니까? 그걸 믿을 수 있습니까? (11p 문구가 대놓고 유발 — 최우선 준비)

**답변**: "앙상블은 모델이 아니라 다섯 모델의 박스를 합친 박스 목록이라, 모델 하나를 받아 채점하는 공식 채점기에 넣을 입구가 없습니다. 그래서 박스 목록끼리 정답과 대조하는 채점기를 만들었고, 공정성을 위해 단일 모델들도 같은 채점기로 다시 재서 비교했습니다. 이 숫자는 절대값 주장이 아니라 같은 잣대 안에서의 상대비교로만 썼고, 발표 내내 공식 잣대의 숫자와 구분해 말씀드리고 있습니다."

**주머니**: 채점 로직 = 확신도 내림차순 그리디 매칭, IoU 0.5, COCO식 AP(코드 스니펫 그림 `concept_charts/scorer_snippet.png`). 코드 머리에 "공식 val 절대값과 다름, 상대비교만 유효" 주석을 처음부터 박아둠.

## Q2. "스크리닝 기준"이 무슨 뜻입니까? conf를 관대하게 잡아 앙상블을 부풀린 것 아닙니까? (11p 문구 유발)

**답변**: "후보를 최대한 쓸어담아 사람이 확인하는 운영 방식을 스크리닝이라고 표현했습니다. 저희도 이 설정이 앙상블에 유리한 건 아닌지 궁금해서 문턱을 올려가며 재채점해봤는데, 문턱을 올리면 격차가 줄고 아주 높은 문턱에서는 단일이 낫습니다. 즉 앙상블의 이득은 스크리닝 운영점의 이득이 맞고, 저희 용도가 정확히 그것입니다. 문턱과 무관한 AP로 봐도 damage 0.743에서 0.811로 앙상블이 앞서고, 단일 모델이 박스를 0개 그린 균열을 앙상블이 잡은 사례는 문턱과 무관합니다."

**주머니**: 문턱 스윕 실측 — 0.05: 0.856 vs 0.894 / 0.25: 동률 / 0.50: 0.562 vs 0.407(역전). WBF의 이득은 저확신 박스들의 상호 확인에서 나옴(낮은 문턱이 측정에 필요) + 단일/앙상블 동일 적용. 정성: DJI_0961.

## Q3. 외부 데이터에서 0.86이 0.06으로 무너졌으면 과적합 아닙니까? (14p가 직접 유발)

**답변**: "과적합이라면 같은 도메인의 새 데이터에서도 무너져야 하는데, 학습에 쓰지 않은 test 301장에서는 0.85를 유지했습니다. 무너진 건 촬영 조건과 라벨 기준이 전혀 다른 외부 데이터였고, 해상도를 맞춰도, 전처리를 바꿔도, 다른 방식의 앙상블로 돌려도 같은 결과라 도메인 차이가 원인이라고 판단했습니다. 그래서 개선 방향도 규제가 아니라 다양한 도메인의 데이터 확보로 잡았습니다."

**주머니**: 과적합 = 같은 유형의 새 문제에서도 무너짐 / 도메인갭 = 시험지 자체가 다른 세상. 학습곡선 train·val loss 동반 하강(정상 수렴) + test301(미학습) 0.85 유지 + 원인규명 3종 표(14p에 있음).

## Q4. rtdetr은 학습이 터졌다면서 왜 앙상블에 넣었습니까? (6p "낙제생"과 11p "에이스"가 유발)

**답변**: "혼자서의 점수와 팀에서의 역할이 다르기 때문입니다. rtdetr은 CNN 계열인 YOLO와 원리가 달라 틀리는 지점도 다릅니다. 앙상블은 서로 다른 실수를 하는 모델들이 빈틈을 메워주는 구조라, 발언권을 절반으로 낮춰 넣었더니 재현율이 가장 높아졌습니다. 발산도 원인을 파봤는데 학습률을 낮추면 시점만 늦춰지고 결국 터져서, 모델 탓이 아니라 저희 GPU 환경의 수치 불안정이 원인이었습니다."

**주머니**: 4모델 0.891 -> +rtdetr 0.944(잣대B). WBF 가중치 [1,1,1,1,0.5]. 발산 실험: lr 0.0008로 낮추자 ep19 -> ep24로 지연만.

## Q5. 왜 최신 모델이 아니라 오래된 yolov8s를 썼습니까? (6p "최신이 항상 낫진 않다"가 유발)

**답변**: "저희도 처음엔 최신이 나을 거라 생각해서, 다섯 모델을 완전히 같은 조건으로 학습해 오디션을 열었습니다. 결과는 yolov8s가 종합 승자였습니다. 짐작이 아니라 숫자로 확인하고 선택한 것입니다. 참고로 YOLO 번호는 만든 곳이 제각각이라 번호가 높다고 더 좋은 게 아니어서, 같은 도구상자 안에서 공정 비교가 가능한 세대들을 골랐습니다."

**주머니**: 5모델 표 — v8s가 mAP50-95 0.627 + damage recall 0.707로 이중 1위. 전부 s 체급·50ep/640/seed42 통일.

## Q6. 정상 이미지를 넣는 게 왜 헛경보를 줄입니까? (9p가 유발)

**답변**: "기본 데이터는 라벨 있는 사진만 써서, 모델이 '모든 사진엔 어딘가 문제가 있다'는 전제로 배웁니다. 그래서 깨끗한 표면에서도 억지로 손상을 찾아낸 겁니다. 정상 사진을 빈 라벨로 넣어주면 '정답 박스가 0개인 경우도 있다'를 배웁니다. 위조지폐를 감별하려면 진짜 지폐도 봐야 하는 것과 같습니다."

**주머니**: 빈 라벨 = 배경 negative(박스 0개가 정답인 지도학습). 헛경보 30 -> 14%, recall은 1500장까지 유지 후 2500장(train의 절반)서 꺾임 -> 최적 750~1500. 헛경보율 측정 = 학습에 안 쓴 정상 200장에 예측 돌려 헛박스 뜬 사진 비율.

## Q7. 오염(dirt)은 외부에서 아예 0이 됐던데, 실무에서 문제 안 됩니까? (14p 오른쪽 그림이 유발)

**답변**: "저희 운영 도메인에서는 오히려 dirt가 가장 잘 잡히는 클래스입니다 — test에서 AP 0.97이었습니다. 외부에서 무너진 건 오염 라벨의 기준이 데이터셋마다 크게 달라서인데, 저희 데이터는 원거리 촬영이라 오염 박스가 화면의 3분의 1을 덮는 관례였고 외부 데이터는 근접 촬영에 박스가 7~10분의 1 크기라, 모델이 배운 것과 아예 다른 물체였던 셈입니다. 다만 안전에 직결되는 건 구조 손상 쪽이라 damage recall 개선을 우선했고, dirt의 일반화는 다양한 도메인 데이터 확보 로드맵으로 남겼습니다."

**주머니**: dirt 박스 중앙값 32%(우리) vs 4.9%(blade30), 크기 순위 역전(14p 그림 그대로 가리키며 답하면 됨).

## Q8. 열화상이 답이라면, 왜 RGB 카메라로 했습니까? (17p TO-BE가 유발)

**답변**: "주어진 데이터가 RGB 드론 사진이었고, RGB로 어디까지 되는지 한계를 확인하는 것 자체가 이번 프로젝트의 결과물입니다. 표면 손상 탐지는 RGB로 충분히 됐고, 표면 아래 손상과 낯선 환경 일반화가 한계로 확인됐습니다. 그 한계가 열화상 같은 다른 센서와 다양한 도메인 데이터라는 다음 단계의 근거가 된 것입니다."

## Q9. 영상 데모와 웹서비스는 정확도가 낮아 보이는데 왜 보여줍니까? (16p "아직 서툼" 문구가 유발)

**답변**: "되는 것과 안 되는 것을 그대로 보여드리는 게 맞다고 생각했습니다. 유튜브 영상은 저희가 배운 데이터와 또 다른 도메인이라, 앞서 보여드린 도메인 차이가 영상에서도 그대로 재현됩니다. 저 데모는 완성품이 아니라 파이프라인이 실제로 돌아간다는 증명이고, 정확도는 다양한 도메인 데이터를 배우면 따라온다는 게 저희 결론입니다."

# 2순위 — 자료(깃허브 문서·차트)를 깊게 본 사람용

## Q10. 슬라이드마다 recall 숫자가 다른데(0.72 근처와 0.94), 뭐가 맞는 겁니까?

슬라이드에서 두 숫자를 일부러 대조하지 않아서(잣대 혼동 방지 설계) 나올 확률은 낮지만, 차트 라벨(0.722)을 읽은 사람은 물을 수 있다.

**답변**: "두 숫자는 채점 잣대가 다릅니다. 0.72는 공식 채점기로 잰 단일 모델 성적이고, 앙상블은 자체 채점기로 동일 조건 비교를 했습니다. 그 잣대에서 단일 최고 0.856이 앙상블 0.944가 된 것이라, 앙상블 효과는 0.856에서 0.944로 읽어주시면 됩니다."

**주머니**: 같은 모델(yolov8s_norm1500)이 잣대A로 0.722, 잣대B로 0.856 — 이 차이가 잣대 차이의 크기.

## Q11. 깃허브 문서를 보니 mAP50-95는 단일이 앙상블보다 높던데요? (깃허브 읽은 사람용)

**답변**: "맞습니다, 위치 정밀도는 단일이 0.005 정도 미세하게 높습니다. WBF가 박스들을 가중평균하며 위치가 살짝 뭉개지고, 앙상블이 새로 찾아낸 어려운 손상일수록 박스가 헐렁하기 때문입니다. 하지만 앙상블의 목적은 놓침을 줄이는 것이었고 recall이 0.856에서 0.944로 올랐습니다. 놓침의 비용이 박스 몇 픽셀 오차보다 크기 때문에 남는 장사라고 판단했습니다."

**주머니**: 단일 yolo11s 0.547 vs 앙상블 0.542(잣대B). 틀 = 인정 -> 산 것 -> 남는 장사.

## Q12. WBF가 뭡니까? NMS와 뭐가 다릅니까?

**답변**: "둘 다 겹친 박스를 정리하는 방법인데, NMS는 겹치면 최고 점수 하나만 남기고 버리고, WBF는 겹친 박스들을 확신도 가중평균으로 융합해 하나로 만듭니다. 여러 모델의 의견을 버리지 않고 합의를 만든다는 점에서 앙상블에 잘 맞아 채택했고, 팀에서 NMS 방식 2모델 앙상블도 병행해 두 방식 모두 효과를 확인했습니다."

## Q13. train/val/test 분할은 어떻게 했고 그 test를 믿을 수 있습니까?

**답변**: "라벨 있는 2,995장을 8대1대1로 나눴는데, 오염이 1대15로 희귀해서 무작위로 자르면 평가가 운에 휘둘립니다. 그래서 오염 포함 여부로 층화해 골고루 배분했고 seed를 고정해 누가 돌려도 같은 분할이 나옵니다. 이후 모든 실험은 학습 재료만 바꾸고 시험지인 test 301장은 한 번도 건드리지 않았습니다."

**주머니**: 정상주입도 train에만(val/test 바이트 동일 검증). 팀 재현 일치(yolov10s 0.846).

## Q14. 이 데이터는 뭐고 어디서 났습니까? 라벨은 누가 단 겁니까? (2p 개요가 유발 — 나올 확률 높은데 뒤늦게 추가해서 번호만 뒤다)

**답변**: "덴마크 공대(DTU)가 공개한 NordTank 풍력터빈 드론 점검 사진으로, 캐글에서 받았습니다. 13,470장에 크기가 586x371로 균일하고, 오염과 손상 라벨이 함께 배포됩니다. 저희가 그대로 믿지 않고 먼저 뜯어봤는데, 라벨 폴더의 labels.txt는 정답 박스가 아니라 클래스 이름표라 학습에서 제외했고, 78%가 라벨 없는 정상 사진이라는 것과 오염 대 손상 박스가 1대 15라는 분포도 이 검수에서 나왔습니다."

**주머니**: EDA 4종 수치(13,470장 / 정상 78% / 1:15 / 박스 크기 32% vs 0.15%)는 통합 노트북에 실제 실행 출력으로 있음.

# 팀원별 갈래 질문 (팀 공유용 — 자기 갈래는 자기에게 물어볼 수 있다)

각자 자기 갈래에서 나올 만한 질문과 답변 뼈대다. 상세 기록은 repo `team/` 폴더의 각자 문서 참고. (답변 뼈대는 취합자 기준 정리라, 본인이 아는 내용으로 살 붙여 말하면 된다.)

### 박정우 — "NMS 앙상블이 뭡니까? WBF와 뭐가 다릅니까?" (10p)

위 Q12와 같은 질문. 뼈대: yolov8s는 기준(conf)을 낮춰 긴가민가한 것까지 넓게 줍는 돋보기, yolo12s는 어텐션으로 확실한 것만 잡는 현미경 — 역할을 나눠 각자 추론하고, 겹친 박스는 NMS로 최고점 하나만 남겨 합쳤다. WBF는 버리는 대신 확신도 가중평균으로 융합한다는 점이 다르고, 팀에서 두 방식 모두 효과를 확인했다.

### 이예령 — "타일화(SAHI)가 뭡니까? 왜 했습니까?" (13p)

뼈대: blade30 원본은 5456 수준 초고해상도라, 640으로 통짜 리사이즈하면 작은 손상이 뭉개진다. 그래서 사진을 960 크기 타일로 쪼개 해상도를 살리는 전처리(SAHI 계열)를 했다. 결과는 "타일화 무죄" — 타일을 해도 안 해도 외부 데이터에선 똑같이 무너져서, 원인이 해상도·전처리가 아니라 도메인 자체임을 격리해준 실험이 됐다.

### 김민 — "증강 재학습은 왜 효과가 없었습니까?" / "영상 데모는 어떻게 만들었습니까?" (12p·16p)

증강 뼈대: albumentations로 손상 이미지를 약 12,000장 규모까지 불리고 정상을 섞어 재학습했다. 배합을 여러 번 바꿔가며(5:5 -> 파인튜닝 -> 증강 90%+정상 10%) 시도했지만 recall 0.7의 벽은 못 넘었고, 어디까지 안 되는지를 확인한 기록으로 남겼다.
영상 뼈대: 프레임에서 터빈 영역(ROI)을 잘라 확대하고, 6모델 앙상블로 탐지한 뒤, 결과를 원본 좌표로 복원한다(zoom ROI auto-follow). 유튜브 영상은 우리가 배운 데이터와 또 다른 도메인이라 오탐이 있고, 그걸 감추지 않고 그대로 보여주는 게 방침.

### 최병철 — "웹서비스는 어떻게 만들었고 뭘 합니까?" (17p)

뼈대: 구글 AI Studio로 만들어 Google Cloud Run에 배포했다. 영상을 올리면 2초 간격으로 프레임을 뽑아 오염·손상을 탐지한다. 현장의 AS-IS(사람이 눈으로 보고 마우스로 수동 박스)에서 한 걸음 나아간 것이고, 표면 아래 손상은 RGB 카메라의 한계라 TO-BE는 열화상 같은 다른 센서 — 우리 도메인갭 결론과 정확히 이어진다.

## 답변 공통 요령

- **걱정형 질문**(문제 안 되냐)엔: 안심(숫자) -> 원인 -> 계획 순서. 원인부터 말하면 "네, 문제입니다"로 들린다.
- **의심형 질문**(부풀린 것 아니냐)엔: 인정할 것 먼저 인정 -> 검증해본 사실 -> 문턱 무관 증거. "저희도 그게 궁금해서 재봤습니다"가 최고의 방패.
- 숫자를 말할 땐 잣대(공식/자체)를 붙인다. 0.72와 0.94를 한 문장에 잇지 않는다.
- 모르는 질문이 오면: "거기까지는 실험을 못 해봤습니다. 방향은 ~로 잡고 있습니다"라고 정직하게.
